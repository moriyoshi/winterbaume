### HTTP protocol pitfalls (HEAD response body)

S3's HEAD operations have a documented quirk that does not surface in the Smithy operation declaration on its own: 4xx responses must not carry a response body. The Smithy model encodes this only obliquely, by giving `HeadBucket` and `HeadObject` the error shape `NotFound` with an empty member set (`{ type: "structure", members: {} }`). If the implementation routes HEAD errors through a generic XML error helper, the SDK silently degrades:

- The HTTP status is correct (404, 403, or 400), but
- `aws-sdk-rust` cannot resolve the response into the typed `HeadBucketError::NotFound` / `HeadObjectError::NotFound` variant because the presence of a body forces the Unhandled branch.
- AWS user-guide reference: [HeadBucket API](https://docs.aws.amazon.com/AmazonS3/latest/API/API_HeadBucket.html) — "A message body isn't included, so you can't determine the exception beyond these HTTP response codes."

Parity rule for winterbaume: HEAD handlers must short-circuit before the generic error-response helper and return `{ status, headers, body: empty }`. Tests must assert both that the body is empty and that the SDK resolves the typed error variant; `assert!(err.is_err())` is not enough to catch this regression. Affected operations are `HeadBucket` and `HeadObject` today; if AWS adds further HEAD operations they inherit the same rule. See [GitHub issue #3](https://github.com/winterbaume/winterbaume/issues/3) for the original bug.

### Conditional-write headers (If-Match / If-None-Match)

S3 models RFC 7232 precondition headers as `@httpHeader`-bearing input members on multiple operations. The Smithy model is the authoritative list — do not infer them from the URL or the body:

| Operation | Members (with `httpHeader`) |
|---|---|
| `PutObject` | `IfMatch` (`If-Match`), `IfNoneMatch` (`If-None-Match`) |
| `CompleteMultipartUpload` | `IfMatch` (`If-Match`), `IfNoneMatch` (`If-None-Match`) |
| `CopyObject` | `IfMatch`, `IfNoneMatch`, plus `CopySourceIfMatch`, `CopySourceIfNoneMatch`, `CopySourceIfModifiedSince`, `CopySourceIfUnmodifiedSince` (all on the destination request) |
| `GetObject` / `HeadObject` | `IfMatch`, `IfNoneMatch`, `IfModifiedSince`, `IfUnmodifiedSince` |
| `DeleteObject` | `IfMatch` (`If-Match`), `IfMatchLastModifiedTime` (`x-amz-if-match-last-modified-time`), `IfMatchSize` (`x-amz-if-match-size`) — conditional delete |
| `RenameObject` | destination `IfMatch`/`IfNoneMatch`/`IfModifiedSince`/`IfUnmodifiedSince`, plus source-side `x-amz-rename-source-if-match`, `x-amz-rename-source-if-none-match`, `x-amz-rename-source-if-modified-since`, `x-amz-rename-source-if-unmodified-since` |
| `UploadPartCopy` | source-side conditional headers analogous to `CopyObject` |

The canonical machine-readable form of the same list lives in the auto-generated `HTTP Bindings` section's "Conditional-write/read coverage" trailer; cross-check that line when adding a new operation in the future.

Semantics (from [Conditional requests](https://docs.aws.amazon.com/AmazonS3/latest/userguide/conditional-requests.html) and the per-operation API references):

- `If-None-Match: *` on PutObject / CompleteMultipartUpload — write only when the destination key does not exist; otherwise return **412 Precondition Failed**. This is the atomic-create primitive used for object-backed locking.
- `If-Match: "<etag>"` on PutObject / CompleteMultipartUpload — write only when the current object's ETag matches; otherwise 412. Compare-and-swap primitive.
- Concurrent operations that race the conditional check may yield **409 ConditionalRequestConflict** instead of success or 412.
- Header values follow RFC 7232 syntax: a quoted ETag (`"abc"`), a list of comma-separated quoted ETags, the weak form `W/"abc"`, or the wildcard `*`. Comparison should be quote-and-weak-stripped.

Parity gotcha: **`PreconditionFailed` is not in `PutObject.errors` in the Smithy model.** PutObject only models `EncryptionTypeMismatch`, `InvalidRequest`, `InvalidWriteOffset`, `TooManyParts`. The `errors:` column of the operation matrix is therefore not a complete picture of which HTTP error codes the operation can emit; the conditional-write errors are documented behaviour without modelled error shapes. Future research must not treat the modelled `errors:` list as exhaustive for any S3 write operation.

Tests must call `.if_match(...)` / `.if_none_match(...)` explicitly and assert `ErrorMetadata.code == "PreconditionFailed"` plus HTTP 412. See [GitHub issue #4](https://github.com/winterbaume/winterbaume/issues/4) for the original bug.

## Official AWS Documentation Research

Sources:
- https://docs.aws.amazon.com/AmazonS3/latest/userguide/mpuoverview.html
- https://docs.aws.amazon.com/AmazonS3/latest/userguide/Versioning.html
- https://docs.aws.amazon.com/AmazonS3/latest/userguide/upload-objects.html
- https://docs.aws.amazon.com/AmazonS3/latest/userguide/conditional-requests.html
- https://docs.aws.amazon.com/AmazonS3/latest/API/API_HeadBucket.html
- https://docs.aws.amazon.com/AmazonS3/latest/API/API_HeadObject.html
- https://docs.aws.amazon.com/AmazonS3/latest/API/API_PutObject.html

Research outcomes:
- Multipart upload is a three-phase lifecycle: initiate, upload numbered parts, then complete or abort. The upload ID is required for part upload, listing, completion, and abort operations.
- Part numbers are in the range 1-10000. Re-uploading the same part number overwrites the earlier uploaded part for that upload.
- Completion concatenates uploaded parts in ascending part-number order. The completed object's ETag is not necessarily an MD5 hash of object bytes.
- In-progress multipart uploads and uploaded parts remain billable until the upload is completed or aborted; aborting makes the upload ID unusable for further part uploads.
- ListParts and ListMultipartUploads return at most 1000 entries per request. AWS documentation explicitly warns not to use ListParts output as the source of truth for CompleteMultipartUpload; clients must submit the part numbers and ETags they recorded.
- Multipart uploads can race with other writes or deletes to the same key. In versioned buckets, completing a multipart upload creates a new version, and the current version is determined by upload initiation time rather than completion time.
- Conditional writes (`If-Match`, `If-None-Match`) on PutObject, CompleteMultipartUpload, and CopyObject are the documented atomic-create / compare-and-swap primitive. Violations return 412 PreconditionFailed; conflicting concurrent operations may return 409 ConditionalRequestConflict. The full member matrix lives under "Behavioural Model Notes -> Conditional-write headers".
- HEAD operations (HeadBucket, HeadObject) return 4xx without a response body. The body's presence is what determines whether aws-sdk-rust resolves the typed `NotFound` variant. See "Behavioural Model Notes -> HTTP protocol pitfalls" for the parity rule.

Parity implications:
- Model multipart uploads as durable in-progress resources with parts, part ETags/checksums, initiation time, metadata, and storage class.
- Preserve versioning-sensitive races and delete-marker interactions separately from simple last-write-wins object storage.
- Implement pagination limits and markers for multipart listings.
- Implement checksum and conditional-write errors separately from generic validation errors.

## Research Checklist for Parity Work

- Confirm lifecycle transitions for every create/update/delete/start/stop operation.
- Confirm exact not-found, already-exists, conflict, validation, throttling, and access-denied error names.
- Confirm pagination token format, result ordering, default limits, and empty collection shape.
- Confirm idempotency-token behaviour, especially mismatched replay parameters.
- Confirm cross-service identifiers such as ARNs, IAM roles, KMS keys, S3 buckets, VPC resources, and logging destinations.
- Confirm whether read APIs are derived from customer-managed state, AWS-managed catalogues, telemetry, recommendations, or asynchronous jobs.
- For every operation, enumerate every modelled input member carrying `@httpHeader`, `@httpQuery`, `@httpPrefixHeaders`, or `@httpPayload`. The hand-written handler signature must list each one, mark it as honoured, intentionally unsupported, or ignored-with-rationale. Conditional headers (`If-Match`, `If-None-Match`, `If-Modified-Since`, `If-Unmodified-Since`, and the `x-amz-copy-source-if-*` variants on copy operations) are the easiest to miss because they don't appear in the URL or body.
- Treat the Smithy `errors:` list as a lower bound, not the full set. S3 documents additional HTTP error codes (notably 412 PreconditionFailed on conditional writes and 409 ConditionalRequestConflict on races) that are not modelled as typed error shapes. Cross-reference each operation's API-reference page for "Errors", "Conditional", "Response" sections.
- For HEAD operations, confirm the 4xx body rule (see "Behavioural Model Notes -> HTTP protocol pitfalls") and confirm tests assert the typed SDK error variant rather than `is_err()` only.
