## Service Overview

Amazon Lightsail is the easiest way to get started with Amazon Web Services (Amazon Web Services) for developers who need to build websites or web applications. It includes everything you need to launch your project quickly - instances (virtual private servers), container services, storage buckets, managed databases, SSD-based block storage, static IP addresses, load balancers, content delivery network (CDN) distributions, DNS management of registered domains, and resource snapshots (backups) - for a low, predictable monthly price. You can manage your Lightsail resources using the Lightsail console, Lightsail API, Command Line Interface (CLI), or SDKs. For more information about Lightsail concepts and tasks, see the Amazon Lightsail Developer Guide. This API Reference provides detailed information about the actions, data types, parameters, and errors of the Lightsail service.

## Official AWS Documentation Research

Sources:
- https://docs.aws.amazon.com/lightsail/latest/userguide/understanding-networking-resources-in-lightsail.html
- https://docs.aws.amazon.com/lightsail/latest/userguide/understanding-lightsail-load-balancers.html
- https://docs.aws.amazon.com/lightsail/latest/userguide/amazon-lightsail-faq-cdn-distributions.html

Research outcomes:
- Lightsail provides simplified resources such as instances, blueprints, bundles, static IPs, disks, snapshots, databases, load balancers, buckets, and CDN distributions.
- Static IPs provide stable addresses that can be detached and reattached across instance restarts or replacements.
- Lightsail load balancers distribute HTTP/HTTPS traffic across instances, perform health checks, support session persistence, and manage SSL/TLS certificates.
- Load balancers are Region-scoped and route only to eligible Lightsail instances.
- CDN distributions accelerate content delivery from origins such as instances, buckets, or load balancers, support HTTPS, and can invalidate cached content.
- Deleting a load balancer does not delete target instances.

Parity implications:
- Model simplified resource families separately instead of mapping everything directly to EC2 or CloudFront state.
- Static IP attach/detach should preserve address identity independently of instance lifecycle.
- Load balancer health, session persistence, certificates, and target attachments should drive routing behaviour.
