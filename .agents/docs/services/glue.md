## Official AWS Documentation Research

Sources:
- https://docs.aws.amazon.com/glue/latest/dg/add-crawler.html
- https://docs.aws.amazon.com/glue/latest/dg/monitor-continuations.html
- https://docs.aws.amazon.com/glue/latest/dg/programming-etl-connect-bookmarks.html

Research outcomes:
- Crawlers run custom classifiers in configured order. The first classifier that recognises data creates the schema; lower-priority classifiers are skipped.
- If no custom classifier matches, built-in classifiers attempt schema inference.
- Crawlers group data into tables or partitions using heuristics, then write metadata to a Data Catalog database.
- Generated table names are restricted to alphanumerics and underscores, can include a custom prefix up to 64 characters, are truncated to 128 characters, and get a hash suffix on collisions.
- Repeated crawler runs look for new or changed files or tables and can add new tables and partitions.
- Job bookmarks track source-processing state per job, version, run number, attempt number, and transformation context.
- Bookmark modes include Enable, Disable, and Pause. Pause processes a selected range without updating bookmark state.
- S3 bookmarks use last-modified timestamps; JDBC bookmarks use sequential primary keys by default or configured bookmark keys.

Parity implications:
- Model Data Catalog databases, tables, partitions, classifiers, crawlers, crawler runs, jobs, job runs, and bookmark state separately.
- Crawler execution should apply ordered classifier selection, table-name generation, partition heuristics, and update/delete behaviour.
- Job run APIs should preserve bookmark mode, run attempts, transformation-context state, reset/rewind operations, and atomic commit of bookmark state.

## Current Network Resource Stub Semantics

Glue currently stores connection networking data as Glue connection properties.

- Connection records can include physical connection requirements such as subnet ID, security group ID lists, and availability zone.
- Jobs, crawlers, and sessions can refer to Glue connections without network reachability checks.
- Glue does not create ENIs or validate subnet and security group combinations.
- The implementation does not consult `winterbaume-ec2` state for these identifiers, so it does not check that referenced VPCs, subnets, security groups, VPC endpoints, network interfaces, or load balancers exist, belong to the same VPC, or are in a usable lifecycle state.

## HTTP Bindings

Per-operation input members that bind to HTTP transport surfaces. Optional members are easy to miss because they do not appear in the operation matrix's Required input column. RFC 7232 conditional headers (`If-Match`, `If-None-Match`, `If-Modified-Since`, `If-Unmodified-Since`) and service-specific modifier headers (`x-amz-*`, `x-amzn-*`) surface here. Every handler must list each binding as honoured, intentionally unsupported, or ignored-with-rationale.

_No `@httpHeader`, `@httpQuery`, `@httpPrefixHeaders`, or `@httpPayload` input members are modelled for this service (typical for `awsJson1_*` protocols, where all input flows through the JSON body)._

## Winterbaume LTM Notes

Sources: .agents/docs/LTM/pluggable-backends-and-query-execution-synthesis.md, .agents/docs/LTM/aws-inter-service-integration-priorities.md, .agents/docs/LTM/cross-service-integration-and-engine-boundaries-synthesis.md.

Mode: full distillation.

### Catalogue Role

- Glue is the durable catalogue source for Athena `DataCatalogType::Glue` paths. Databases and tables in `winterbaume-glue` state need to become schema inputs to Athena query execution.
- Athena's current Glue catalogue handling is metadata-only if it stops at `CreateDataCatalog` and `GetDataCatalog`. The missing service behaviour is resolving Glue databases and tables at query time.
- Glue catalogue parity should preserve both Glue CRUD semantics and downstream Athena lookup semantics. A table that exists in Glue but cannot be resolved by Athena is not a complete cross-service implementation.

### Test Implications

- Glue catalogue work should be tested through Athena query execution as well as Glue CRUD. The durable parity question is whether a Glue-managed schema can drive SQL execution.
- High-value negative cases are missing database, missing table, incompatible schema, and dropped-or-renamed Glue resources referenced by an Athena query.
- If the query backend is DuckDB-backed, seed data and schema through the same backend path Athena will use, then verify that Glue state is the source of the schema selected for a Glue-type catalogue.

## Research Checklist for Parity Work

- Confirm lifecycle transitions for every create/update/delete/start/stop operation.
- Confirm exact not-found, already-exists, conflict, validation, throttling, and access-denied error names.
- Confirm pagination token format, result ordering, default limits, and empty collection shape.
- Confirm idempotency-token behaviour, especially mismatched replay parameters.
- Confirm cross-service identifiers such as ARNs, IAM roles, KMS keys, S3 buckets, VPC resources, and logging destinations.
- Confirm whether read APIs are derived from customer-managed state, AWS-managed catalogues, telemetry, recommendations, or asynchronous jobs.
