# Amazon Simple Email Service

Source: AWS Smithy API model vendored in `vendor/api-models-aws`.

## Service Overview

Amazon SES API v2 Amazon SES is an Amazon Web Services service that you can use to send email messages to your customers. If you're new to Amazon SES API v2, you might find it helpful to review the Amazon Simple Email Service Developer Guide. The Amazon SES Developer Guide provides information and code samples that demonstrate how to use Amazon SES API v2 features programmatically.

## v1/v2 State Coherence

- **Paired with `ses` ( same SDK slug `ses`, same endpoint prefix `email`, same ARN namespace `ses` ).** In real AWS, SES v1 ( `awsQuery` ) and SESv2 ( `restJson1`, this crate ) are two protocol surfaces over the same backend. Resources created via one API are observable and mutable via the other.
- AWS docs explicitly note shared management: configuration sets created via v1 `CreateConfigurationSet` are listed by `aws sesv2 list-configuration-sets`; verified identities, templates, suppression list entries, dedicated IP pools, and account-level sending settings are likewise shared.
- **Current Winterbaume status: divergent.** `winterbaume-sesv2` owns its own `SesState` ( `EmailIdentity`, `ConfigurationSet`, `EmailTemplate`, suppression list, dedicated IP pools, contact lists, import/export jobs, multi-region endpoints, tenants, deliverability test reports, reputation entities ); `winterbaume-ses` owns a parallel `SesV1State` ( `Identity`, `ConfigurationSet`, `Template`, `ReceiptRuleSet`, `SentEmail` ). Neither crate depends on the other, so resources created via v2 are invisible to v1 and vice versa — a behavioural gap relative to real AWS.
- **What needs to change:** the shared resource families ( identities, configuration sets, templates, suppression list, account-level settings ) must live in a single state owner — typically a shared inner struct exposed by one crate and consumed by the other, or an extracted common state crate. Receipt rule sets are v1-only ( no v2 equivalent ); the v2-only families listed above stay here. Cross-API integration tests that exercise create-via-v2 / read-via-v1 ( and the reverse ) are the right regression guard.
