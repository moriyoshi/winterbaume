# AWS IoT

Source: AWS Smithy API model vendored in `vendor/api-models-aws`.

## Service Overview

IoT IoT provides secure, bi-directional communication between Internet-connected devices (such as sensors, actuators, embedded devices, or smart appliances) and the Amazon Web Services cloud. You can discover your custom IoT-Data endpoint to communicate with, configure rules for data processing and integration with other services, organize resources associated with each device (Registry), configure logging, and create and manage policies and credentials to authenticate devices. The service endpoints that expose this API are listed in Amazon Web Services IoT Core Endpoints and Quotas. You must use the endpoint for the region that has the resources you want to access. The service name used by Amazon Web Services Signature Version 4 to sign the request is: execute-api .

## Official AWS Documentation Research

Sources:
- https://docs.aws.amazon.com/iot/latest/developerguide/what-is-aws-iot.html
- https://docs.aws.amazon.com/iot/latest/developerguide/mqtt.html
- https://docs.aws.amazon.com/iot/latest/developerguide/iot-device-shadows.html

Research outcomes:
- AWS IoT Core connects devices and applications through MQTT, MQTT over WebSocket, HTTPS, and device SDKs.
- Things represent devices in the registry and can have thing types, attributes, groups, certificates, and policies associated with them.
- X.509 certificates and IoT policies authorise device connections and topic operations independently from IAM control-plane permissions.
- MQTT supports QoS levels, retained messages, persistent sessions, last-will messages, shared subscriptions, MQTT 5 properties, and connection management features with AWS-specific differences.
- Device shadows store desired and reported device state as JSON documents and publish delta messages when desired and reported state diverge.
- Named and unnamed shadows allow multiple independent state documents per thing.
- Rules route messages from MQTT topics to AWS service actions.

Parity implications:
- Model things, thing groups, thing types, certificates, IoT policies, policy attachments, MQTT sessions, retained messages, shadows, rules, jobs, and endpoints separately.
- Data-plane authorisation should evaluate IoT policies and certificate state rather than IAM alone.
- Shadow update/version/delta behaviour should be stateful and topic-driven.

## HTTP Bindings

Per-operation input members that bind to HTTP transport surfaces. Optional members are easy to miss because they do not appear in the operation matrix's Required input column. Every handler must list each binding as honoured, intentionally unsupported, or ignored-with-rationale.
