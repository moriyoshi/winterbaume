# Terraform Converter Codegen and Resource Coverage

## Summary

The Terraform converter layer now uses a spec-driven code generator for Terraform state resource models, with generated serde projections isolated in `winterbaume-tfstate-resource-models`. Converter maintenance is a split responsibility: specs and generated models handle flat Terraform attributes, while hand-written converter bodies keep ownership of state mutation, nested-block reshaping, warning-only resource handling, and registration. Coverage reporting now combines per-service resource coverage with per-resource attribute coverage and is regenerated by the `update-readme` skill.

## Key Facts

- `tools/tf-converter-codegen/` owns generated `*TfModel` structs from TOML specs.
- Generated model files live in `crates/winterbaume-tfstate-resource-models/src/`, not inside `winterbaume-terraform`.
- `winterbaume-terraform` re-exports the generated crate as `crate::generated`, so converter imports keep the stable `use crate::generated::<svc> as <svc>_gen;` shape.
- Most specs use `mode = "model_only"`; converters deserialize typed models with `serde_json::from_value` and then hand-write state projection.
- The spec type vocabulary is intentionally small: `string`, `string?`, `u32`, `i64`, `bool`, and `tags`. Nested blocks, `Vec<T>`, arbitrary maps, and discriminated shapes stay hand-written.
- `tf-converter-codegen ingest <service>` is useful for bootstrapping specs but requires human review for view names, required fields, renames, resource splits, and numeric types.
- Resource coverage is tracked by `.agents/docs/TERRAFORM_RESOURCE_COVERAGE.md`; attribute coverage is tracked by `.agents/docs/TERRAFORM_CONVERTER_COVERAGE.md`.
- `update-readme` regenerates Terraform reports, `docs/reference/terraform.md`, the services reference footer, service crate READMEs, and `docs/services/` pages together.
- Large converter waves should let sub-agents work on per-service spec/generated/converter files and merge `winterbaume-server/src/main.rs` registrations centrally.
- Warning-only converters are acceptable when Terraform must parse a resource type but the backing service state does not model the family yet.

## Details

### Spec-Driven Converter Models

The 2026-05 migration replaced hand-repeated Terraform attribute unpacking with a codegen layer analogous to `tools/smithy-codegen/`. The generator is deliberately simple: `clap`, `anyhow`, string concatenation, `rustfmt`, and no `quote!`, `syn`, or external templating. It exposes `gen`, `gen-all`, `check`, and `list`.

The chosen architecture is thin projection through serde. A per-service TOML spec emits `*TfModel` structs with `Serialize` and `Deserialize`, field-level `rename`, `default`, and `skip_serializing_if`, and Rust keyword handling via raw identifiers such as `r#type`. Converter call sites then do:

```rust
let model: svc_gen::SomeResourceTfModel = serde_json::from_value(attrs.clone())?;
```

The converter remains responsible for:

- `TerraformResourceConverter` trait scaffolding
- ARN and URL templates beyond simple field projection
- multi-resource composition
- state snapshot, merge, and restore orchestration
- nested-block conversion
- warning-only resource semantics
- `depends_on_types`

This keeps non-trivial mappings visible to reviewers instead of hiding them in generated output.

### `winterbaume-tfstate-resource-models`

The generated model tree was moved out of `crates/winterbaume-terraform/src/generated/` into a new leaf crate:

```text
crates/winterbaume-tfstate-resource-models/
  Cargo.toml
  src/lib.rs
  src/<service>.rs
```

The new crate depends only on `serde`. This avoids recompiling thousands of generated serde lines whenever hand-written converter code changes. `winterbaume-terraform/src/lib.rs` keeps compatibility with existing converter imports:

```rust
pub use winterbaume_tfstate_resource_models as generated;
```

SQS was the only thin-projection blocker because its generated file referenced both `ConversionContext` and `QueueStateView`, which would have created a dependency cycle after extraction. SQS was converted to `mode = "model_only"` and moved the `into_state_view` / `From<&QueueStateView>` logic into `converters/sqs.rs`.

Future spec edits should run:

```bash
./.agents/bin/cargo.sh run -p tf-converter-codegen -- gen <service>
./.agents/bin/cargo.sh run -p tf-converter-codegen -- check
```

The default output path already points at `crates/winterbaume-tfstate-resource-models/src/`.

### Coverage Tooling

Two Terraform coverage reports answer different questions:

- `generate_terraform_resource_coverage.py`: which Terraform AWS provider resource types are registered for each Winterbaume service.
- `generate_terraform_converter_coverage.py`: which Terraform schema attributes are consumed on inject and emitted on extract by each converter.

Both scripts share `.agents/skills/api-coverage/scripts/tf_schema.py`, which fetches and caches the AWS provider schema under `.agents-workspace/tmp/tf-schema/aws_provider_schema.json`. The cache loader accepts either the inner `resource_schemas` dict or the full `terraform providers schema -json` envelope.

`update-readme/scripts/update_readme.py` now runs both Terraform reports unless `--no-terraform` is passed. It also updates `docs/reference/terraform.md` and threads the live distinct resource count into `docs/reference/services.md` instead of using a hard-coded resource total.

### Coverage Heuristic Fixes

The first regenerated Terraform reference looked like a regression because the attribute parser could not see newer converter idioms. The fixes were:

- Add `kinesisanalyticsv2` to `PREFIX_OVERRIDES` with both `aws_kinesis_analytics_` and `aws_kinesisanalyticsv2_`.
- Render override prefixes as literals, for example `` `aws_vpc`, `aws_subnet`, +35 more `` instead of `override (37 patterns)`.
- Index generated `*TfModel` structs and credit their serde field names to inject coverage when converter bodies reference those models.
- Scan macro invocation bodies for `"aws_..."` resource-type literals so warning-only and S3 macro-generated converters appear in the per-resource table.

After the fixes, the per-resource table rose from 1026 detected converters to 1143, and the spec-vs-doc audit found zero spec-declared resource types missing from the attribute coverage table.

### Resource Expansion Waves

The first post-codegen coverage wave added high-impact service coverage:

- IAM: OIDC, SAML, MFA, certificates, credential resources, password policy, aliases, inline policies, policy attachments, and exclusive attachment resources.
- S3: bucket sub-resources and named configurations such as versioning, ACL, logging, lifecycle, replication, encryption, public access block, notification, CORS, website, analytics, metrics, object lock, and intelligent tiering.
- Route53, API Gateway, RDS, Redshift, Glue, and SageMaker gained many additional resource converters.

EC2 then received a dedicated sequential six-wave expansion because all work touched the same spec, generated model, converter file, and server registration file. The EC2 wave added 100 resource types, raising EC2 from 100 missing to 0 missing in the resource coverage report.

A later parallel tail sweep added roughly 603 converters across about 50 services. The working pattern was:

1. Dispatch one sub-agent per service for independent spec/generated/converter files.
2. Require returned `injector.register(...)` snippets.
3. Merge `winterbaume-server/src/main.rs` centrally.
4. Run the cargo gate from the orchestrator after each batch.

By the end of the sweep, handled Terraform resource types rose from 430 to 1127, with 77 classified missing resources remaining.

### Warning-Only Converters

Warning-only converters parse a Terraform resource model and claim the resource type, but do not mutate service state and return empty extraction. They are appropriate when:

- the service crate does not model the resource family
- implementing real state would be a larger service-layer task
- Terraform state still needs to parse without reporting an unknown resource

They should be treated as honest placeholders, not as full fidelity.

### Registration and Service Injection

Resource converters are not active until registered in `winterbaume-server/src/main.rs`. Some services had to be hoisted into `InjectableServices` because they previously only existed as inline `MockService` entries. The repeated pattern is:

1. Add an `Arc<winterbaume_<svc>::Service>` field to `InjectableServices`.
2. Create a local `Arc` binding.
3. Replace the inline `Arc::new(...)` mock-service registration with `Arc::clone(&svc) as Arc<dyn MockService>`.
4. Add the field initialiser.
5. Import and register the Terraform converters.

This was needed for services such as pinpoint, servicecatalog, fsx, neptune, memorydb, and sagemaker.

## Pitfalls

- Do not edit generated `crates/winterbaume-tfstate-resource-models/src/*.rs` files directly. Edit the spec and regenerate.
- Do not expect `tf-converter-codegen ingest` output to be review-ready; it is a bootstrap tool.
- Do not expand the spec type vocabulary casually. The small vocabulary is what keeps complicated mappings in hand-written code.
- Do not let sub-agents edit `winterbaume-server/src/main.rs` concurrently during broad converter waves; merge registrations centrally.
- Do not trust sub-agent cargo results from concurrent runs as final. The orchestrator-side post-merge gate is authoritative.
- Do not use `let mut sv = StateView::default(); sv.x = Some(v);` under clippy `-D warnings`; use `StateView { x: Some(v), ..Default::default() }`.
- Do not iterate maps as `for (_, value) in &map`; use `map.values()`.
- Do not treat warning-only converters as completed state support.
- Do not trust missing Terraform coverage rows until prefix overrides, macro parsing, and generated-model heuristic coverage have been checked.
- Do not hand-edit generated `docs/reference/terraform.md` or `.agents/docs/TERRAFORM_*_COVERAGE.md`; fix scripts/specs and regenerate.
