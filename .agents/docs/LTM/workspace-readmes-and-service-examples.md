# Workspace READMEs and Service Examples

## Summary

Winterbaume's documentation and example surface is generated and maintained at workspace scale. The durable knowledge is which scripts own coverage reporting, where runnable examples belong, how service names map to AWS SDK crate names, how floci and kumo now fit into the coverage story, and which crate-renaming changes must be reflected in repo tooling rather than only Cargo manifests.

## Key Facts

- API coverage is generated by `.agents/skills/api-coverage/scripts/generate_coverage.py`.
- README files are refreshed by `.agents/skills/update-readme/scripts/update_readme.py`.
- `CRATE_DISPLAY_INFO` in `update_readme.py` is intentionally sorted alphabetically by crate-name key; resort the table mechanically when adding entries.
- Per-service README regeneration preserves manually-authored sections when the existing file uses `<!-- BEGIN AUTO ... -->` / `<!-- END AUTO -->` markers.
- The supported example layout is the root package's `examples/` directory, not a separate examples crate.
- Coverage reporting now spans Winterbaume, moto, floci, kumo, SDK integration tests, and Terraform E2E coverage.
- The workspace no longer uses `winterbaume-stubs`; supported-service reporting is derived from real service crates and coverage-mapping tables.
- Coverage tooling should stay self-contained by skill. Do not create runtime imports between skills just to share helpers.
- AWS SDK crate naming is not always a mechanical transformation of the Winterbaume crate suffix.
- The 13-crate AWS SDK naming-alignment batch was completed on 2026-04-05, but the mapping knowledge still matters for tooling, docs, and future service additions.

## Details

### Coverage and README Regeneration Workflow

The workspace documentation flow is:

1. regenerate `.agents/docs/API_COVERAGE.md`
2. update the root `README.md`
3. regenerate all per-crate `README.md` files

The ownership split is:

- `generate_coverage.py` computes the report
- `update_readme.py` consumes the report and regenerates docs

The durable rule is that README regeneration follows coverage regeneration. Do not hand-edit generated service-coverage summaries.
Since the 2026-04-18 removal of `winterbaume-stubs`, this workflow assumes every supported service is represented by a real workspace crate rather than a fallback stub registry.

### Preserving Manual README Sections

The 2026-04-25 S3 Terraform note exposed a documentation ownership problem: per-service README files are generated, but some crates still need stable manually-authored sections.

`update_readme.py` now supports splicing generated content into an existing README:

- generated service README content is wrapped in `<!-- BEGIN AUTO ... -->` and `<!-- END AUTO -->`
- `update_service_readmes()` reads the existing file first
- `_splice_auto_block()` replaces only the generated block when both markers exist
- content outside the markers is preserved
- first-time generation without markers keeps the old behaviour and writes the generated content as-is

The durable rule is that durable manual guidance, such as provider configuration notes, belongs outside the generated block. Generated coverage tables and operation summaries still belong inside it.

### Dossier Section Transcription

`update_readme.py` can transcribe selected sections from `.agents/docs/services/<model-slug>.md` into generated service crate READMEs. The mapping uses the API coverage row's Smithy model slug, so the README for `winterbaume-<crate>` reads the matching service dossier by model name rather than guessing from the crate suffix.

The first transcribed section is `## Current Network Resource Stub Semantics`, added to 54 network-aware service dossiers. The regeneration path wrote matching sections into 54 service crate READMEs and also preserved existing example embedding, including `winterbaume-s3files/README.md`.

When adding another transcribed dossier section:

- keep the source of truth in `.agents/docs/services/<model-slug>.md`
- update `.agents/skills/update-readme/SKILL.md` with the generator contract
- run `update_readme.py --no-docs`
- verify the dossier-section count matches the generated README-section count
- trim unrelated newline-only README churn before reporting the result

### Server Access Logging

`winterbaume-server` now emits structured per-request access logs via `tracing::info!`.

Logged fields:

- `remote_addr`
- `method`
- `path`
- `status`
- `elapsed_ms`

The implementation records `Instant::now()` at request entry, preserves method and path across `req.into_body()`, and passes the accepted `remote_addr` into the handler. The durable choice is structured tracing fields rather than a free-form access-log string, so switching to JSON output later does not require log parsing changes.

### Coverage Dimensions Now Tracked

The report surface has widened beyond Winterbaume versus moto.

It now tracks:

- Winterbaume implementation coverage
- moto coverage
- floci coverage
- kumo coverage
- SDK integration-test coverage
- Terraform E2E coverage

That matters because different gaps now mean different things:

- Winterbaume below moto is a service-implementation gap
- Winterbaume above moto but below Terraform coverage targets is a validation gap
- floci and kumo provide cross-emulator context rather than the main parity target

### floci and kumo Integration

The 2026-04-12 coverage work added floci and kumo as comparison columns.

Durable implementation rules:

- local vendor clones are preferred when present
- otherwise the skill can use cached remote fetch results
- remote cache entries should record both fetch date and upstream version
- service-name mapping tables are part of the coverage system, not incidental script detail

This means coverage maintenance now includes four mapping surfaces:

- Winterbaume crate-to-model mapping
- moto model mapping
- floci stem-to-model mapping
- kumo directory-to-model mapping

### Retired Stub-Crate Logic

The old stub-crate reporting path is historical rather than active guidance.

Before 2026-04-18, the coverage and README scripts carried special handling for services routed through `winterbaume-stubs`, including stub-only rows, stub-to-model mappings, and deduplication against real crate rows. That logic existed to keep the supported-services table honest while some moto-backed services still resolved through a 501 fallback crate.

That is no longer the workspace model. `winterbaume-stubs` was removed once an audit showed the remaining entries were dead code and all supported services already had dedicated crates.

The durable rules now are:

- coverage and supported-services reporting should derive from real workspace crates plus explicit emulator mapping tables
- if a service row is missing, duplicated, or shows placeholder metadata, inspect crate-display tables, crate-to-model mappings, and report parser logic before assuming missing stub metadata
- keep per-handler `STUB[...]` annotations conceptually separate from service discovery and README generation; they still matter for handler implementation and quality-gate work, but not for crate registration

### Integration-Test and Terraform Coverage Reporting

The report now includes both SDK integration-test and Terraform E2E coverage.

This created two durable maintenance rules:

- `update_readme.py` must parse the detailed report permissively, not assume a brittle fixed shape
- missing coverage in the report may be a mapping or parser problem before it is a test-suite problem

When a service's Terraform or integration tests seem absent from `.agents/docs/API_COVERAGE.md`, check the coverage script's mapping tables and parsing logic before assuming the service lacks tests.

### Example Layout and Conventions

The steady-state example layout is the root `examples/` directory.

Per-service examples should:

1. build a `MockAws` with the relevant service
2. configure the AWS SDK client with the mock transport and credentials
3. call one representative list, describe, or get operation when practical
4. print a small summary

If a service needs extra runtime context, the example should still show correct client setup and explain what is missing rather than inventing a fake live call.

### CRATE_TO_MODEL Drift and Orphan Cleanup

`/api-coverage` and `/update-readme` reported 164 services on 2026-04-27 even though `crates/winterbaume-*` contained 211 service crates ( 226 total minus 15 infra/utility ). Root cause: the coverage script counts only crates listed in `CRATE_TO_MODEL` inside `.agents/skills/api-coverage/scripts/generate_coverage.py`, and that table had drifted behind the workspace.

The fix had three parts:

1. **47 missing crate -> model mappings added to `CRATE_TO_MODEL`.** Several model dirs needed dash-vs-no-dash care, e.g. `applicationdiscoveryservice` -> `application-discovery-service`, `pcaconnectorscep` -> `pca-connector-scep`, `route53recoverycluster` -> `route53-recovery-cluster`, `cloudfrontkeyvaluestore` -> `cloudfront-keyvaluestore`, `cloudtraildata` -> `cloudtrail-data`, `cloudsearchdomain` -> `cloudsearch-domain`. Cross-reference `vendor/api-models-aws/models/` to verify each name.
2. **Extended `_DOCS_SERVICE_SKIP`** in `.agents/skills/update-readme/scripts/update_readme.py` with non-service crates whose READMEs were leaking into `docs/services/`: `bedrock-flow-parser`, `bedrock-flow-validator`, `iam-rule-eval`, `sfn-asl-eval`, `wafv2-wcu-calculator`, `wafv2-webacl-rule-parser`.
3. **Orphan cleanup in `generate_docs_service_pages()`.** The function now tracks the set of files it wrote and `unlink()`s any other `*.md` under `docs/services/`. This removed 13 stale pages from old crate names that had been renamed since: `apigatewaymanagementapi.md`, `cognitoidp.md`, `dms.md`, `ds.md`, `elb.md`, `elbv2.md`, `events.md`, `lex.md`, `logs.md`, `meteringmarketplace.md`, `sdb.md`, `sesv1.md`, `stepfunctions.md`. Without this, renames silently leave dead pages in the VitePress site.

After the fix the three counts agree ( the canary that the system is consistent ): 211 service crates reported by coverage, 211 service-crate `README.md` files refreshed, 211 `docs/services/*.md` pages.

Recurring lessons:

- The coverage script's "services" count comes purely from `CRATE_TO_MODEL`. The presence of a crate directory or even a `handlers.rs` file is not enough -- you must register the crate in the mapping or it is silently invisible to every downstream report. Worth keeping in mind whenever `add-service` produces a new crate.
- The README script's `CRATE_DISPLAY_INFO` table is sorted alphabetically by crate suffix. If an entry is appended out of order, resort the whole dict in one mechanical pass and confirm the script still parses, e.g. by running its `--help` path.
- The update-readme script and the coverage script use *different* discovery mechanisms. update-readme walks `crates/winterbaume-*/README.md` directly, so it can produce more docs pages than the coverage script knows about. Verifying the three counts agree is the canary; treat any mismatch as a sign that one of the mappings has drifted.
- Renames are append-only as far as `docs/services/` is concerned, unless the generator explicitly cleans up. The orphan-cleanup loop now fixes this for future renames too.

### Crate Rename Maintenance

The rename from `winterbaume-tagging` to `winterbaume-resourcegroupstagging` established the durable rule: crate renames are not complete when Cargo builds again.

They also require updates in:

- workspace members and dependencies
- feature names and re-exports
- server wiring
- examples and integration tests
- coverage and README scripts
- any hard-coded crate-to-model or display-name tables

The 13-crate AWS SDK alignment batch reinforced that rule at workspace scale.

### Rename procedure: `winterbaume-databasemigrationservice` → `winterbaume-databasemigration` ( 2026-04-28 )

The DMS crate was named `winterbaume-databasemigrationservice` but the published AWS SDK Rust client is `aws-sdk-databasemigration` ( without the trailing `service` ). The mismatch traced back to a v0.0.0 placeholder crate `aws-sdk-databasemigrationservice` that was mistakenly chosen during the original implementation; that placeholder also propagated an inaccurate "no real Rust DMS SDK exists" note into the QG-8.1 TODO, the README, the docs, and three LTM documents.

The full rename touched:

- Directory: `crates/winterbaume-databasemigrationservice/` → `crates/winterbaume-databasemigration/` ( via `git mv` )
- Package name in the crate's `Cargo.toml`
- Workspace `Cargo.toml`: members list ( appears twice ), `workspace.dependencies` entry, and the root package's dev-dependencies on both the winterbaume crate and the AWS SDK crate ( `aws-sdk-databasemigrationservice = "*"` → `aws-sdk-databasemigration = "1"` )
- Consumers: `winterbaume-server` ( Cargo.toml + main.rs ), `winterbaume-terraform` ( Cargo.toml + tests/integration_test.rs + src/converters/dms.rs ), `examples/dms.rs`
- Crate's own `README.md` and `tests/{integration,scenario}_test.rs` ( `winterbaume_databasemigrationservice` → `winterbaume_databasemigration` )
- Top-level `README.md`, `docs/services/databasemigrationservice.md` ( renamed to `docs/services/databasemigration.md` ), `docs/reference/services.md`
- `.agents/docs/API_COVERAGE.md` ( three rows ), `.agents/docs/TODO.md` ( both the active and historical entries, with cross-references back to the old name for traceability )
- Skill scripts: `.agents/skills/api-coverage/scripts/generate_coverage.py` ( crate-to-model map ) and `.agents/skills/update-readme/scripts/update_readme.py` ( crate-to-name map, example-slug alias, server-mode CLI map, and the `_CRATE_NOTES` placeholder entry — note rewritten to point users at the real `aws-sdk-databasemigration` SDK )
- LTM documents `repo-maintenance-and-agent-workflows-synthesis.md`, `crate-publishing-and-release.md`, `github-issue-triage-and-automation.md` — placeholder claims rewritten to clarify the slug confusion

Files intentionally left unchanged:

- `.github/ISSUE_TEMPLATE/{bug_report,feature_request}.yml` already used the correct `databasemigration` slug ( the original LTM note in `github-issue-triage-and-automation.md` already documented this divergence ).
- `.agents-workspace/tmp/qg-results/databasemigrationservice.md` — frozen quality-gate snapshot from 2026-04-27, preserved as historical artefact.
- The 2026-04-22 historical TODO checkbox keeps a parenthetical pointer back to the old crate name for grep-ability.

Verification: `cargo fmt -p winterbaume-databasemigration -p winterbaume-server -p winterbaume-terraform -p winterbaume`, then `cargo build -p winterbaume-databasemigration` ( succeeds in 3m 26s clean compile ). `Cargo.lock` resolves cleanly with 4 references to `winterbaume-databasemigration` ( none to the old name ) and `aws-sdk-databasemigration v1.110.0` pulled in. The placeholder `aws-sdk-databasemigrationservice` v0.0.0 is no longer in the dependency graph.

Recurring lesson: when naming a new `winterbaume-<service>` crate, cross-check the published `aws-sdk-<slug>` name on crates.io rather than assuming the Smithy model dir name is authoritative. v0.0.0 placeholder crates can be misleading — always verify version > 0.

### `CRATE_DISPLAY_INFO` Protocol Coverage ( 2026-04-28 )

The Supported Services table in `README.md` had `?` in the **Protocol** column for 47 crates. Root cause is `update_readme.py:513-514`:

```python
display_name, protocol = CRATE_DISPLAY_INFO.get(
    crate, (crate.removeprefix("winterbaume-"), "?")
)
```

`CRATE_DISPLAY_INFO` ( declared at `update_readme.py:22` ) is a hand-maintained dict mapping crate → `(display_name, protocol)`. When a crate is missing from the dict, the fallback yields the lowercase short name and a literal `?`. So every `?` row corresponded to a crate that had not been added to the dict since it landed in the workspace — not a code-detection failure, just a stale lookup table.

Fix procedure: determine the correct protocol for each missing crate by reading the `aws.protocols#...` trait directly from each service's Smithy model under `vendor/api-models-aws/models/`. Distribution for the 47 added entries:

- **REST-JSON ( 35 ):** accessanalyzer, aiops, amplifybackend, amplifyuibuilder, appconfigdata, appfabric, appflow, appintegrations, applicationcostprofiler, applicationsignals, arczonalshift, artifact, backupsearch, cloudfrontkeyvaluestore, cloudsearchdomain, cloudtraildata, dlm, fis, outposts, pcaconnectorscep, personalizeevents, personalizeruntime, pinpointsmsvoice, rbin, rolesanywhere, s3outposts, savingsplans, simspaceweaver, snowdevicemanagement, ssmquicksetup, supportapp, taxsettings, trustedadvisor.
- **awsJson1.0 ( 7 ):** backupgateway, bcmdashboards, bcmrecommendedactions, cloudcontrol, freetier, keyspaces, route53recoverycluster.
- **awsJson1.1 ( 6 ):** applicationdiscoveryservice, applicationinsights, autoscalingplans, bcmdataexports, cloud9, pi, pricing.

Note: the model directory name is not always the same as the `winterbaume-<x>` suffix ( e.g. `applicationdiscoveryservice` → `application-discovery-service`, `arczonalshift` → `arc-zonal-shift`, `pcaconnectorscep` → `pca-connector-scep`, `route53recoverycluster` → `route53-recovery-cluster`, `autoscalingplans` → `auto-scaling-plans` ). Mirrors the existing slug convention captured in long-term memory for `aws-sdk-rust`.

Long-term improvement worth pursuing: derive `(display_name, protocol)` from the Smithy model at generation time rather than maintaining a parallel hand-edited dict. The protocol trait is already authoritative in `vendor/api-models-aws/`; only the human display name needs hand-curation. Tracked in TODO.md as `crate-display-info-derive`. The dict's "Keep sorted alphabetically by crate name" comment ( line 21 ) is aspirational — the dict is in fact append-ordered. New batch was appended at the end to match the existing convention rather than fight it; tracked as `crate-display-info-resort` in TODO.md if a future cleanup wants to enforce the comment.

The later `crate-display-info-derive` fix implemented that long-term improvement:

- `generate_coverage.py` owns `derive_protocol(model_name)`, which reads the Smithy service protocol trait from `vendor/api-models-aws/models/<model>/service/*/*.json`.
- `update_readme.py` now keeps only human display names in `CRATE_DISPLAY_INFO`; protocol values are derived from `CRATE_TO_MODEL`.
- `CRATE_DISPLAY_INFO_LEGACY` is empty and exists only as an escape hatch for a future crate without a Smithy model entry.
- The refactor corrected historical protocol drift: `cloudwatch` is `awsJson1.0`, `ec2` is `ec2Query`, `ec2instanceconnect` is `awsJson1.1`, and `s3control` is `restXml`.

After large operation expansions, rerun the coverage and README workflow so public tables reflect new operation counts.

### Service Dossier Usage Scenarios

Service dossiers under `.agents/docs/services/` now carry `Possible Usage Scenarios` sections as test-planning input. The broad EC2-derived scenario archetypes were propagated only when operation and resource names exposed concrete cues such as associations, attachments, memberships, target groups, account defaults, jobs, executions, tasks, deployments, streams, workspaces, clusters, channels, meetings, simulations, or migration tasks.

Scenario bullets describe useful coverage candidates and expected read-side observations. They are not claims that the current Winterbaume crate already implements the full scenario.

### Example Limits

Some services still cannot demonstrate a meaningful live call in a minimal example because the mock needs extra runtime context.

That is acceptable. The example is still useful if it:

- shows correct mock and client wiring
- documents the missing prerequisites
- does not fabricate a misleading API call

## Pitfalls

- Do not create a separate examples workspace crate when the root package can host `examples/`.
- Do not assume AWS SDK crate names match Winterbaume crate suffixes mechanically.
- Do not treat missing report rows as proof that tests do not exist before checking mapping tables.
- Do not look for `winterbaume-stubs` when debugging supported-services tables; that crate and its stub-row pipeline were retired.
- Do not update Cargo manifests for a rename without also updating the scripts and docs that depend on the old name.
- Do not invent fake live example calls for services that need extra runtime context.
- Do not put manual provider-usage notes inside generated README blocks; they will be replaced on regeneration.
- Do not duplicate dossier-derived guidance by hand in service crate READMEs. Add or update the dossier section and let `update_readme.py` transcribe it.
- Do not parse access logs from free-form text when structured tracing fields are available.
- Do not pick a `winterbaume-<service>` crate name from the Smithy model directory alone. Cross-check the published `aws-sdk-<slug>` name on crates.io and verify the version is > 0.0.0; v0.0.0 placeholders can be misleading and will silently propagate the wrong slug into READMEs, docs, TODO entries, and LTM documents.
- Do not assume `?` in the README Protocol column means coverage tooling cannot detect the protocol. It almost always means a missing entry in `CRATE_DISPLAY_INFO`. Read the `aws.protocols#...` trait directly from `vendor/api-models-aws/models/<service>/service.json` to determine the right value, then append the entry.
- Do not hand-maintain protocol labels when the Smithy model has the authoritative trait.
- Do not treat service-dossier scenarios as implementation guarantees. They are planning prompts.
